1. OOP & Thiết kế lớp

Dễ

Hệ thống có những lớp nào? Hãy mô tả cây kế thừa (inheritance hierarchy).

Hệ thống có hai cây kế thừa chính:

Cây User:

Entity (abstract, id)
  └── User (abstract: username, password, email, isActive)
        ├── Bidder  — balance, deductBalance()
        ├── Seller  — storeName, rating
        └── Admin   — accessLevel

Cây Item:

Entity (abstract, id)
  └── Item (abstract: name, description, startingPrice, imagePath)
        ├── Art         — artist, theme, material
        ├── Vehicle     — year, mileage
        └── Electronics — brand, warrantyMonths

Ngoài ra còn các lớp độc lập: Auction, BidTransaction, AutoBid, AuctionStatus (enum).

Dễ

Tại sao lớp Item lại là abstract? Phương thức abstract nào nó khai báo?

Item là abstract vì không có "item chung chung" trong thực tế — mỗi item phải thuộc một danh mục cụ thể (Art, Vehicle, Electronics). Việc abstract hóa ép các subclass phải cài đặt logic riêng của mình.

3 phương thức abstract:

  • getCategoryDetails() — trả về thông tin đặc trưng của từng loại (ví dụ Art trả về artist/theme/material)
  • insertSubData(Connection conn, int itemId) — chèn dữ liệu vào bảng con (art_items, vehicle_items, electronics_items)
  • updateSubData(Connection conn) — cập nhật dữ liệu bảng con

Đây là ví dụ của Template Method Pattern: ItemDAO.insert() gọi chung item.insertSubData() mà không cần biết đây là Art hay Vehicle.

Dễ

Encapsulation được áp dụng như thế nào trong project?

Encapsulation được áp dụng nhất quán:

  • Tất cả fields đều là private hoặc protected — không truy cập trực tiếp từ ngoài
  • Truy cập qua getter/setter được kiểm soát (ví dụ Bidder.deductBalance() kiểm tra số dư trước khi trừ)
  • Auction.lock (ReentrantLock) là private final, chỉ expose qua getLock()
  • SessionManager dùng Singleton — hide constructor, chỉ expose getInstance()
  • DAO classes ẩn toàn bộ SQL queries khỏi business logic
Trung bình

Polymorphism được sử dụng ở đâu trong project này?

Ít nhất 4 điểm sử dụng polymorphism:

  1. ItemDAO.insert(item): Tham số kiểu Item nhưng runtime có thể là Art, Vehicle, hoặc Electronics. Gọi item.insertSubData() — mỗi subclass tự xử lý bảng của mình
  2. ItemFactory.createItem(): Returns Item nhưng thực tế tạo ra subclass tương ứng với type string
  3. AuctionDetailController.fillUI(): Dùng item.getClass().getSimpleName() để xác định category label mà không cần instanceof
  4. User hierarchy: UserDAO trả về User nhưng runtime là Bidder/Seller/Admin với behavior khác nhau
Trung bình

Lớp Entity có vai trò gì? Tại sao cần lớp base này?

Entity là abstract base class chung cho tất cả domain objects, chứa field id và getter/setter tương ứng.

Lý do cần Entity:

  • DRY (Don't Repeat Yourself): Mọi object trong hệ thống đều có ID — không cần viết lại ở từng class
  • DAO có thể dùng entity.getId() một cách thống nhất
  • Là nền tảng cho các framework ORM nếu mở rộng sau này
Trung bình

Tại sao Bidder có method deductBalance(amount) thay vì chỉ setter setBalance()?

Đây là nguyên tắc Tell, Don't Ask — đối tượng tự quản lý state của mình.

deductBalance(amount):

  • Kiểm tra balance >= amount trước khi trừ
  • Nếu không đủ → trả về false (không throw exception ngay)
  • Ngăn code bên ngoài vô tình set balance âm bằng cách gọi setBalance(balance - amount) mà không check

So sánh: setBalance(-100) sẽ không có validation nào → dễ bug. deductBalance() đảm bảo invariant "balance không âm".

Trung bình

Factory Pattern được hiện thực như thế nào trong ItemFactory? Tại sao cần Factory?

ItemFactory.createItem(type, name, desc, price, attributes):

switch (type) {
  case "ART"         → new Art(artist, theme, material)
  case "VEHICLE"     → new Vehicle(year, mileage)
  case "ELECTRONICS" → new Electronics(brand, warrantyMonths)
  default            → new Electronics(...)   // fallback
}

Attributes là Map<String, String> chứa các thuộc tính đặc trưng (brand cho Electronics, artist cho Art...).

Tại sao cần Factory:

  • Tách logic tạo object khỏi code sử dụng — client code không cần biết Art hay Electronics có gì
  • Khi thêm loại item mới (ví dụ "Jewelry"), chỉ sửa Factory, không phải sửa tất cả nơi tạo Item
  • Cả server handler lẫn controller đều gọi ItemFactory.createItem() — code nhất quán
Trung bình

Singleton Pattern được dùng ở đâu? Hiện thực theo cách nào? Có thread-safe không?

4 Singleton trong project:

  • AuctionScheduler — quản lý timer đóng auction tự động
  • AuctionNotifier — broadcast events đến observers
  • SessionManager — lưu thông tin user đang đăng nhập
  • ThemeManager — trạng thái Dark/Light mode

Pattern hiện thực (Lazy Initialization):

private static AuctionScheduler instance;

public static synchronized AuctionScheduler getInstance() {
    if (instance == null) instance = new AuctionScheduler();
    return instance;
}

Thread safety: Dùng synchronized trên method getInstance() → thread-safe nhưng có overhead. Trong JavaFX app, SessionManager chỉ được gọi từ JavaFX Application Thread nên không cần synchronized.

Trung bình

Observer Pattern được hiện thực như thế nào? Cho ví dụ cụ thể một notification flow.

Interfaces:

interface Subject { addObserver(o); removeObserver(o); notifyObservers(msg); }
interface Observer { update(message); }

AuctionNotifier implements Subject, lưu danh sách observers trong CopyOnWriteArrayList (thread-safe).

Ví dụ flow khi có bid mới:

  1. BiddingService.placeBid() thành công
  2. Gọi AuctionNotifier.getInstance().notifyNewBid(auctionId, 1500.0, "alice")
  3. Notifier format message: "NEW_BID|42|1500.0|alice"
  4. Loop qua tất cả registered observers → gọi observer.update(message)
  5. Mỗi observer tự parse và xử lý (ví dụ UI update, log...)

Lưu ý: Trong implementation hiện tại, Observer pattern dùng để log/notify internal events. Không phải WebSocket — client không nhận real-time push mà phải reload.

Khó

DAO Pattern giải quyết vấn đề gì? Nếu không có DAO, code sẽ trông như thế nào?

DAO (Data Access Object) tách biệt business logic khỏi data persistence.

Không có DAO:

// Trong BiddingService.placeBid() (business logic)
PreparedStatement ps = conn.prepareStatement(
    "UPDATE auction SET current_price=?, highest_bidder_id=? WHERE id=?");
ps.setDouble(1, newPrice);
// SQL lẫn lộn với business logic ❌

Với DAO:

// BiddingService chỉ gọi:
auctionDAO.updateBid(conn, auctionId, newPrice, bidderId); // ✅
// SQL nằm trong AuctionDAO, hoàn toàn ẩn

Lợi ích của DAO trong project:

  • Đổi từ MySQL sang PostgreSQL → chỉ sửa DAO classes
  • Business logic classes có thể test bằng mock DAO
  • Code dễ đọc hơn: auctionDAO.findById(id) thay vì SQL dài

Trung bình

Database transaction được dùng ở đâu? Tại sao cần transaction?

Transaction được dùng trong 2 nơi quan trọng nhất:

1. BiddingService.placeBid():

conn.setAutoCommit(false);
try {
    // Cập nhật auction.current_price và highest_bidder_id
    // Insert bid_transaction record
    // Process auto-bids
    // Anti-sniping check
    conn.commit();
} catch (Exception e) {
    conn.rollback(); // Hoàn tác toàn bộ nếu bất kỳ bước nào fail
}

2. PaymentService.processPayment():

// Trừ balance bidder
// Cộng 90% cho seller
// Cộng 10% cho admin
// Update auction.status = PAID
// Tất cả trong 1 transaction

Tại sao cần: Nếu không có transaction và server crash sau khi trừ tiền bidder nhưng trước khi cộng tiền seller → tiền biến mất. Transaction đảm bảo "all or nothing".

Trung bình

Mapper classes (AuctionMapper, BidTransactionMapper) làm gì? Có thể bỏ chúng được không?

Mapper classes convert ResultSet (raw JDBC data) thành Java objects:

// AuctionMapper.map(ResultSet rs) → Auction
Auction a = new Auction();
a.setId(rs.getInt("id"));
a.setCurrentPrice(rs.getDouble("current_price"));
a.setStatus(AuctionStatus.valueOf(rs.getString("status")));
// ...

Có thể bỏ không? Về lý thuyết có, nhưng khi đó code mapping sẽ nằm rải rác trong từng DAO method. Mapper:

  • Tách concern: DAO chỉ lo query, Mapper lo convert
  • Cùng query có thể dùng nhiều mapper khác nhau (ví dụ: full object vs lightweight DTO)
  • Dễ test mapper độc lập với mock ResultSet
Khó

SQL injection được phòng chống như thế nào?

Toàn bộ project dùng PreparedStatement — không bao giờ concatenate user input vào SQL string.

// ❌ Vulnerable (KHÔNG làm thế này):
String sql = "SELECT * FROM users WHERE username = '" + username + "'";

// ✅ An toàn (cách project làm):
PreparedStatement ps = conn.prepareStatement(
    "SELECT * FROM users WHERE username = ?");
ps.setString(1, username); // ? được parameterized, không thể inject

PreparedStatement tự động escape ký tự đặc biệt (', --, ;...) trước khi gửi xuống DB engine. Kể cả khi user nhập '; DROP TABLE users; --, nó sẽ được xử lý như một string literal bình thường.

4. Kiến trúc Client-Server

Dễ

Tại sao project lại dùng kiến trúc Client-Server? Trước đây không có kiến trúc này?

Trước đây: JavaFX controller trực tiếp gọi DAO → kết nối MySQL từ UI thread. Vấn đề:

  • Database credentials hardcoded hoặc trong config file phía client
  • Network calls (JDBC) trực tiếp từ UI → có thể block UI thread
  • Không có tầng trung gian để validate/authorize
  • Bid safety: race condition khi 2 user bid cùng lúc từ 2 máy khác nhau

Sau khi thêm Client-Server:

  • Server tập trung xử lý business logic và DB access
  • Bid được serialize qua ReentrantLock trên server — thread-safe
  • Client (JavaFX) chỉ gọi REST API qua HTTP — UI không block do Task<T>
  • Dễ scale: có thể deploy server riêng, nhiều client connect vào
Dễ

HTTP server được tạo ra như thế nào? Tại sao không dùng Spring Boot hay Tomcat?

Dùng JDK HttpServer (com.sun.net.httpserver) — thư viện built-in của Java, không cần dependency ngoài:

HttpServer server = HttpServer.create(new InetSocketAddress(8080), 0);
server.createContext("/auction/list",  new AuctionHandler());
server.createContext("/auth/login",    new AuthHandler());
// ...
server.setExecutor(Executors.newCachedThreadPool());
server.start();

Tại sao không Spring Boot:

  • Đây là đồ án học thuật (Lập trình Nâng cao) — mục tiêu là hiểu cách hoạt động của HTTP, không phải dùng framework
  • Spring Boot thêm hàng trăm MB dependencies cho một project nhỏ
  • JDK HttpServer đủ mạnh cho use case của project
  • Hiện thực thủ công giúp hiểu rõ HTTP protocol hơn
Trung bình

Dữ liệu được truyền giữa client và server như thế nào? Format gì?

Dùng JSON qua Gson 2.11.0.

Phía Client (ApiClient):

    .uri(URI.create(serverUrl + path))
    .GET().build();
// Parse response: Gson.fromJson(body, JsonObject.class)

// Đọc body:
String body = new String(exchange.getRequestBody().readAllBytes());

// Trả về:
byte[] response = JsonUtil.toJson(result).getBytes(UTF_8);

Ví dụ response từ /auction/bid:

{ "success": true, "newPrice": 1500.00 }
Trung bình

AUCTION_CACHE trong AuctionHandler là gì? Tại sao cần cache?

private static final ConcurrentHashMap<Integer, Auction> AUCTION_CACHE = new ConcurrentHashMap<>();

Mục đích: Lưu Auction objects trong memory để share ReentrantLock giữa các request.

Tại sao phải có: BiddingService dùng auction.getLock() để serialize bids. Lock này phải là cùng một object cho mọi request đến cùng auction. Nếu mỗi request tạo Auction mới từ DB → mỗi request có lock khác nhau → mất thread safety!

// getCachedAuction() dùng computeIfAbsent:
AUCTION_CACHE.computeIfAbsent(id, k -> {
    // Chỉ load từ DB lần đầu tiên
    return auctionService.getById(conn, k);
});

Lưu ý: Cache này là in-memory per-JVM. Khi server restart, cache bị clear → load lại từ DB. Đây là acceptable cho single-instance deployment.

Khó

Điều gì xảy ra nếu client mất kết nối giữa chừng khi đang đặt bid?

Kịch bản: Client gửi POST /auction/bid → server nhận → đang xử lý → client timeout → TCP connection đóng.

Phía server: Server tiếp tục xử lý request bình thường. Transaction được commit. Bid được lưu vào DB. Khi server cố ghi response → IOException (broken pipe) → catch và log. Server không rollback chỉ vì client ngắt.

Phía client: HttpClient throw HttpTimeoutException (sau 15 giây) → Task.setOnFailed() → hiển thị error message "Lỗi kết nối".

Vấn đề: Client thấy lỗi nhưng bid đã được lưu. Đây là vấn đề của "at-least-once delivery" trong distributed systems.

Cách xử lý hoàn chỉnh hơn (nếu có thời gian): Return idempotency key với mỗi bid request → server check xem bid này đã xử lý chưa trước khi thực hiện.

5. Concurrency & Thread Safety

Trung bình

Race condition có thể xảy ra ở đâu trong hệ thống đấu giá? Được giải quyết như thế nào?

Kịch bản race condition: Alice và Bob đặt bid cùng lúc cho auction #42 với current_price = 1000.

Thread A (Alice's bid $1100):      Thread B (Bob's bid $1050):
  read currentPrice = 1000           read currentPrice = 1000
  1100 > 1000 ✓ validate OK          1050 > 1000 ✓ validate OK
  UPDATE current_price = 1100        UPDATE current_price = 1050  ← lost update!
  INSERT bid_transaction Alice $1100 INSERT bid_transaction Bob $1050

Giải pháp: ReentrantLock trên mỗi Auction object:

auction.getLock().lock(); // chỉ 1 thread được phép vào tại một thời điểm
try {
    // Validate + Update + Insert — atomic
} finally {
    auction.getLock().unlock();
}

Bob phải chờ Alice xong mới vào. Khi đó currentPrice đã là 1100, bid $1050 của Bob bị từ chối (InvalidBidException).

Trung bình

Tại sao AuctionNotifier dùng CopyOnWriteArrayList thay vì ArrayList?

ArrayList không thread-safe: nếu một thread đang iterate để notify observers, thread khác add/remove observer → ConcurrentModificationException.

CopyOnWriteArrayList: mỗi write operation (add/remove) tạo một copy mới của array → iteration luôn đọc snapshot tại thời điểm bắt đầu → an toàn concurrent reads.

Trade-off: Write chậm hơn (phải copy), nhưng trong use case này:

  • Observer đăng ký/hủy: hiếm (chỉ khi mở/đóng popup)
  • Notify: thường xuyên (mỗi bid)
  • → Optimize cho đọc: CopyOnWriteArrayList là lựa chọn đúng
Trung bình

JavaFX Task<T> và Platform.runLater() được dùng để làm gì?

JavaFX có quy tắc: chỉ được update UI từ JavaFX Application Thread (JAT).

Vấn đề: Gọi HTTP (ApiClient) blocking → nếu gọi từ JAT → UI freeze trong khi chờ server.

Giải pháp:

Task<Double> task = new Task<>() {
    @Override protected Double call() throws Exception {
        // Chạy trên BACKGROUND thread — safe để gọi HTTP
        JsonObject resp = ApiClient.post("/auction/bid", body);
        return resp.get("newPrice").getAsDouble();
    }
};
task.setOnSucceeded(e -> Platform.runLater(() -> {
    // Chạy trên JavaFX Application Thread — safe để update UI
    priceLabel.setText(String.format("%,.2f USD", task.getValue()));
}));
new Thread(task).start();

Platform.runLater() enqueue một Runnable vào JAT's event queue → được execute trên đúng thread → không crash.

Khó

AuctionScheduler hoạt động như thế nào? Tại sao dùng daemon thread?

ScheduledExecutorService scheduler =
    Executors.newSingleThreadScheduledExecutor(r -> {
        Thread t = new Thread(r);
        t.setDaemon(true); // ← quan trọng!
        return t;
    });

scheduler.scheduleAtFixedRate(
    this::closeExpiredAuctions, // task
    0,    // initial delay
    30,   // period
    TimeUnit.SECONDS
);

closeExpiredAuctions(): Mỗi 30s, query tất cả auction có status = OPEN/RUNNING và endTime < NOW → gọi endAuction() + processPayment() cho từng cái.

Daemon thread: Khi main thread (JavaFX app) thoát, JVM chỉ dừng nếu không còn non-daemon thread nào đang chạy. Nếu scheduler là non-daemon → JVM không thoát được khi user đóng app. Daemon thread tự động bị kill khi JVM shutdown.

6. Nghiệp vụ Đấu giá

Dễ

Một auction trải qua những trạng thái (status) nào? Vẽ state diagram.

                   Admin approve
   [PENDING] ──────────────────────→ [OPEN]
      │                                  │ Scheduled/endTime passed
      │ Admin reject                     ↓
      ↓                              [RUNNING]
   [REJECTED]                            │ endTime passed OR manual end
                                         ↓
                                     [FINISHED]
                                         │ Payment processed
                                         ↓
                                      [PAID]

   [CANCELED] ← Seller/Admin cancel (from OPEN or RUNNING)

Chi tiết:

  • PENDING: Vừa tạo, chờ admin duyệt
  • OPEN: Admin duyệt, sẵn sàng nhận bid (OPEN và RUNNING đều cho phép bid)
  • RUNNING: Đang diễn ra (có thể là after first bid)
  • FINISHED: endTime qua, đang chờ payment
  • PAID: Payment đã được xử lý
  • REJECTED: Admin từ chối kèm lý do
  • CANCELED: Đã hủy
Trung bình

Auto-bid hoạt động như thế nào? Cho ví dụ cụ thể với số liệu.

Setup: Auction #42, current_price = $1000. Alice đăng ký auto-bid: maxBid=$1500, increment=$100.

Kịch bản: Bob đặt bid thủ công $1100.

Flow trong processAutoBids():

1. Bob bid $1100 → current_price = $1100
2. Sau khi commit bid của Bob, gọi processAutoBids()
3. Load auto-bid của Alice: max=$1500, increment=$100
4. nextBid = 1100 + 100 = $1200
5. 1200 ≤ 1500 → Alice auto-bids $1200
6. current_price = $1200
7. Kiểm tra lại: có auto-bid nào khác không? Không → dừng
8. Response trả về newPrice = $1200

Nếu Bob đặt $1600: nextBid = 1600 + 100 = $1700 > maxBid $1500 → Alice không auto-bid, Bob thắng.

Priority: Nhiều auto-bid → sort theo createdAt (FIFO) — đăng ký sớm hơn ưu tiên hơn.

Trung bình

Anti-sniping là gì? Tại sao cần tính năng này?

Sniping: Kỹ thuật đặt bid vào giây cuối cùng (ví dụ 3 giây trước khi hết) để không ai có thời gian counter-bid.

Anti-sniping trong project:

private void applyAntiSniping(Connection conn, Auction auction) {
    long secsLeft = Duration.between(LocalDateTime.now(), auction.getEndTime()).getSeconds();
    if (secsLeft <= 10) {
        // Gia hạn thêm 60 giây
        LocalDateTime newEnd = auction.getEndTime().plusSeconds(60);
        // UPDATE auction SET end_time = newEnd WHERE id = ?
        auction.setEndTime(newEnd);
    }
}

Nếu ai đó bid trong 10 giây cuối → auction tự động kéo dài thêm 60 giây → các bidder khác có thêm thời gian để counter-bid. Đây là tính năng phổ biến trên các nền tảng đấu giá thực tế như eBay.

Trung bình

Payment được xử lý như thế nào? Phân chia tiền ra sao?

PaymentService.processPayment(conn, auction):

  1. Kiểm tra auction.status == FINISHED
  2. Nếu không có highest bidder → đóng auction (không cần payment)
  3. Load bidder từ DB → check bidder.getBalance() >= finalPrice
  4. Nếu không đủ → throw InsufficientBalanceException
  5. Tính toán: adminFee = price * 0.10, sellerAmount = price * 0.90
  6. UPDATE: bidder.balance -= price
  7. UPDATE: seller.balance += sellerAmount
  8. UPDATE: admin.balance += adminFee
  9. UPDATE: auction.status = PAID
  10. Tất cả trong 1 transaction → commit hoặc rollback

Commission model: Admin lấy 10%, Seller nhận 90%. Đây là mô hình phổ biến của marketplaces.

Đây là business rule cơ bản để tránh shill bidding — hành vi gian lận trong đấu giá khi seller tự bid vào item của mình để đẩy giá lên.

if (auction.getSeller() != null &&

Nếu có 1000 user cùng bid vào một auction trong 1 giây, hệ thống có chịu được không?

Về lý thuyết, hệ thống đúng về mặt correctness nhưng sẽ bị bottleneck:

1. Lock contention: 1000 request đến AuctionHandler.handleBid() cùng lúc. HttpServer dùng CachedThreadPool nên tạo 1000 threads. Tất cả cùng gọi auction.getLock().lock() → 999 threads phải wait trong queue của ReentrantLock. Chỉ 1 thread xử lý tại một thời điểm.

2. Database pressure: Mỗi bid = 1 transaction + multiple SQL queries. 1000 bid/s sẽ overload MySQL.

3. Memory: 1000 threads × ~512KB stack = ~500MB chỉ cho threads.

Cải tiến nếu cần scale:

  • Dùng connection pool (HikariCP) thay vì tạo connection mới mỗi request
  • Queue-based bid processing (Kafka) — serialize bids qua message queue
  • Rate limiting per user
  • Virtual threads (Java 21+) thay vì platform threads

Với đồ án học thuật, single-user hoặc tens of users đồng thời là hoàn toàn ổn.

Khó

So sánh ReentrantLock với synchronized. Tại sao chọn ReentrantLock?

FeaturesynchronizedReentrantLock
Trylock (không block)❌✅ tryLock(timeout)
Interruptible❌✅ lockInterruptibly()
FairnessKhông đảm bảo✅ new ReentrantLock(true)
Multiple conditions1 (wait/notify)✅ Nhiều Condition
Code verbosityĐơn giản hơnPhải unlock trong finally

Tại sao project chọn ReentrantLock:

  • Auction.getLock() là instance lock trên mỗi Auction object — mỗi auction có lock riêng, concurrent bidding trên DIFFERENT auctions không block nhau
  • Với synchronized, phải sync trên auction object hoặc class — ít linh hoạt hơn
  • Trong tương lai có thể dùng tryLock(100, MILLISECONDS) để từ chối bid nếu server quá tải
Khó

Tại sao server trả JSON thay vì trả Java objects trực tiếp qua serialization?

Java serialization (ObjectOutputStream) có vấn đề:

  • Tight coupling: client và server phải có CÙNG version của class. Thay đổi field nào → break compatibility
  • Security: Java deserialization có nhiều CVE nổi tiếng (arbitrary code execution)
  • Language-specific: Chỉ Java-to-Java, không thể test bằng browser/Postman
  • Debug khó: Binary format, không đọc được bằng mắt thường

JSON (với Gson):

  • Human-readable → dễ debug với Postman, browser DevTools
  • Language agnostic → frontend web sau này có thể kết nối
  • Decoupled: server có thể thay đổi internal classes mà không break client miễn JSON structure giữ nguyên
  • Industry standard cho REST APIs
Khó

Hãy nêu 3 điểm yếu của hệ thống và đề xuất cải thiện.

1. Không có Authentication token / Authorization

  • Hiện tại: Client gửi userId trong request body → không có gì verify userId đó có phải của user đang login không
  • Cải thiện: Implement JWT (JSON Web Token). Server trả JWT khi login, client gửi kèm mọi request trong Authorization header

2. Real-time chưa thực sự real-time

  • Hiện tại: Client phải reload để thấy bid mới (không có push notification)
  • Cải thiện: WebSocket hoặc Server-Sent Events — server push bid updates tới tất cả connected clients ngay khi có bid mới

3. Không có connection pooling

  • Hiện tại: Mỗi request tạo mới DatabaseConnection.getConnection() → overhead
  • Cải thiện: HikariCP — pool 10-20 connections, tái sử dụng thay vì tạo mới mỗi request

Nếu bạn được thêm tính năng "Đặt giá tối thiểu bí mật" (Reserve Price), bạn sẽ thiết kế như thế nào?

Reserve Price: Seller đặt giá tối thiểu bí mật. Auction chỉ chốt bán nếu highest bid >= reserve price.

Thiết kế:

DB: Thêm column reserve_price DECIMAL(10,2) NULL vào bảng auction.

// Auction.java
private Double reservePrice; // null = không có reserve
public boolean isReserveMet() {
    return reservePrice == null || currentPrice >= reservePrice;
}

UI Seller (AddListingController): Thêm optional field "Reserve Price".

UI Bidder (AuctionDetailController): Hiển thị "Reserve not met" nếu !auction.isReserveMet().

// PaymentService.processPayment():
if (!auction.isReserveMet()) {
    // Không bán, return tiền cho bidder
    auction.setStatus(CANCELED);
    return;
}
// Tiếp tục process payment bình thường
